## 一、初始设置（只做一次，当前电脑已完成）

### 1.1 在 GitHub 上 Fork

打开 https://github.com/agentscope-ai/CoPaw ，点击右上角 **Fork** 按钮。
（已完成，Fork 地址：https://github.com/happinessoverdue/CoPaw ）

### 1.2 克隆你的 Fork 到本地

```bash
# 例如在lilinxin的MacbookM3Max上
cd /Users/lilinxin/CoPaw-src 
git clone https://github.com/happinessoverdue/CoPaw.git CoPaw-dev
cd CoPaw-dev
```

### 1.3 设置官方仓库为上游

```bash
git remote add upstream https://github.com/agentscope-ai/CoPaw.git
```

### 1.4 验证 remote 配置

```bash
git remote -v
```

应该看到：

```
origin    https://github.com/happinessoverdue/CoPaw.git (fetch)
origin    https://github.com/happinessoverdue/CoPaw.git (push)
upstream  https://github.com/agentscope-ai/CoPaw.git (fetch)
upstream  https://github.com/agentscope-ai/CoPaw.git (push)
```

### 1.5 创建你的开发分支

```bash
git checkout -b ampere
```

从此以后，你所有的代码修改都在 `ampere` 分支上做。

## 二、日常开发

在 `ampere` 分支上正常改代码、提交：

```bash
# 确认当前在你的分支上
git branch

# 修改代码后，提交
git add .
git commit -m "描述你改了什么"

# 推送到你的 GitHub Fork（可选，用于备份）
git push origin ampere
```

**重要原则：永远不要在 main 分支上修改代码。**

## 三、官方更新了，如何同步（常用操作）

当你发现官方 CoPaw 有了新的更新，执行以下 5 步：

```bash
# 第 1 步：拉取官方最新代码（不会改你的文件）
git fetch upstream

# 第 2 步：切换到 main 分支
git checkout main

# 第 3 步：把官方更新合入 main（因为 main 没有你的改动，这步是无风险的）
git merge upstream/main

# 第 4 步：切回你的开发分支
git checkout ampere

# 第 5 步：把你的改动变基到最新的 main 之上
git rebase main
```

### rebase 的工作方式

rebase 是**逐个 commit** 处理的，不是一次性合并整个分支：

1. 拿你的第一个 commit，尝试应用到最新 main 上 → 没冲突就继续
2. 拿第二个 commit → 没冲突就继续
3. 以此类推，直到所有 commit 都处理完

所以每个功能单独 commit 很重要——冲突时定位更精准，更容易处理。

**关于多次变基**：每次 rebase 完成后，你的分支就已经基于新的基点了。
下次再变基时，Git 只处理上次基点到新基点之间的差异，不会重复之前的工作。

### 如果 rebase 出现冲突

说明你改的某个文件，官方也改了同一个位置。Git 会暂停并标记冲突文件：

```
<<<<<<< HEAD
官方的代码
=======
你的代码
>>>>>>> your-commit
```

**处理方式**：

- **只保留官方的**：删掉你的代码和三行标记
- **只保留你的**：删掉官方代码和三行标记
- **两边都保留**：删掉三行标记，把两段代码都留着，调整好顺序和逻辑
- 不是所有改动都能简单拼接，比如两边都改了同一个函数签名，需要手动合成

改完之后：

```bash
git add <冲突文件>
git rebase --continue
```

如果改乱了想全部撤销，回到 rebase 之前的状态：

```bash
git rebase --abort
```

**要点**：已经成功处理的 commit 不受影响，你只需要处理当前冲突的那个 commit。
Git 不管你最终怎么改代码，只要 add + continue，它就认为冲突已解决。
最终代码合不合理，靠你自己判断。

### 同步完成后（可选）：推送到 GitHub Fork

```bash
git push origin main            # 让 Fork 上的 main 也保持最新
git push origin ampere    # 让 Fork 上的开发分支也同步
```

如果 rebase 后 push 被拒绝（因为历史改变了），需要强制推送：

```bash
git push origin ampere --force-with-lease
```

| 查看提交历史       | `git log --oneline -10` ,`git log origin/ampere..ampere --oneline #本地尚未push的commit` |

| 查看关联的云端仓库    | `git remote -v`                                                                                 |

### 什么是 remote（远程仓库）

remote 就是"云端仓库的别名"。本地可以关联多个远程仓库：


| 别名         | 指向             | 用途           |
| ---------- | -------------- | ------------ |
| `origin`   | 你的 GitHub Fork | 可拉可推（你的代码备份） |
| `upstream` | 官方仓库           | 只拉不推（没有写权限）  |


- `git remote -v` — 查看所有关联的云端仓库
- 虽然 upstream 会显示 push 地址，但你实际推不上去（没权限），不用担心

### 什么是分支（branch）

分支**不是**文件夹。整个 `CoPaw-dev` 目录同一时刻只显示一个分支的内容。
切换分支就像切电视频道——文件夹不变，里面的文件内容跟着变。


| 分支             | 内容          | 能改吗               |
| -------------- | ----------- | ----------------- |
| `main`         | 纯官方代码       | **不改**，专门用来同步官方更新 |
| `ampere` | 官方代码 + 你的改动 | **在这里改**，日常开发分支   |


- `git branch` — 查看所有分支，带 `*` 的是当前所在分支
- `git checkout <分支名>` — 切换到已有分支
- `git checkout -b <新分支名>` — 创建新分支并切换过去（`-b` = branch，只在创建时用）
- `git branch -vv` — 查看每个本地分支对应的云端分支（跟踪关系）

### commit vs push

- `**git commit`** — 在**本地**保存一次记录（快照），不会上传任何东西。时间戳在 commit 时写死。
- `**git push`** — 把本地的 commit **搬运**到云端（GitHub），不改 commit 的内容和时间。

简单说：commit 是存档，push 是上传存档。

### merge（合并）

把两个分支的内容汇合到一起。如果只有一边有新提交（比如 main 没改过，官方有新提交），
merge 就只是简单地"快进"（fast-forward），不产生额外操作。

我们用 merge 的场景：`git merge upstream/main` — 把官方最新代码合入本地 main。
因为 main 永远不手动修改，所以这步永远是无风险的快进。

### rebase（变基）

把你的提交"搬到"最新的官方代码之上。**只改你的分支，main 完全不变。**

```
rebase 之前：
main:          ──①──②──③──④──⑤  （官方最新）
                    \
ampere:       ──A──B       （你的改动，还在旧版本上）

rebase 之后：
main:          ──①──②──③──④──⑤  （没变）
                                  \
ampere:                     ──A'──B' （你的改动，移到最新版本上了）
```

效果：你的分支始终保持"官方最新 + 你的几个提交"这种干净结构。

### Fork

Fork ≠ 自动同步。Fork 只在点击按钮的那一刻复制一份官方仓库到你账号下，
之后两边独立。官方更新了你的 Fork 不会自动变，需要手动同步（见第三节）。

GitHub 规则：Fork 公开仓库得到的 Fork **一定也是公开的**，无法设为私有。

### git reflog

查看你在本地仓库的操作记录（checkout、commit、rebase 等），最上面是最新的。
不包含 status、diff 这类只读命令。

```bash
git reflog          # 查看 Git 操作记录
history | grep git  # 查看终端里所有 git 命令历史（更全）
```
