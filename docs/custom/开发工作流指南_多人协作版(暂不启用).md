# CoPaw 多人协作开发工作流指南

> 适用场景：多名开发者在官方 CoPaw 基础上共同进行定制化开发，同时持续跟进官方更新。
>
> 本文档是单人版工作流（`docs/custom/开发工作流指南.md`）的多人协作升级版。
>
> **当前状态：暂不启用**，待有同事加入开发时再正式采用。

---

## 总览：从单人到多人，变了什么？

单人开发时，所有改动直接提交到 `ampere` 分支。多人协作后，核心变化是引入 **Feature Branch（功能分支）+ PR（Pull Request）** 机制：

| | 单人模式 | 多人协作模式 |
|---|---|---|
| **ampere 的角色** | 个人开发分支，直接在上面改代码 | 团队集成分支，只通过 PR 合入代码 |
| **开发方式** | 直接 commit 到 ampere | 从 ampere 拉功能分支，开发完提 PR 合并回来 |
| **代码审查** | 无 | 通过 GitHub PR 审查后再合并 |
| **官方同步** | 自己一个人做 rebase | 由维护者（linxin）统一负责 |

### 分支结构全景

```
官方仓库 (upstream)
agentscope-ai/CoPaw
        │
        │  git fetch upstream
        ▼
┌─────────────────────────────────────────────────────────────┐
│  Fork 仓库 (origin): happinessoverdue/CoPaw                │
│                                                             │
│  main ──────────────────────────────────────                │
│    │        纯官方代码，只同步不修改                            │
│    │                                                        │
│    ▼                                                        │
│  ampere ──────────────────────────                    │
│    │   │   │     团队集成分支，通过 PR 合入代码                  │
│    │   │   │                                                │
│    ▼   ▼   ▼                                                │
│  feat/ fix/ feat/   ← 各开发者的功能分支（临时，合并后删除）      │
│  A     B    C                                               │
└─────────────────────────────────────────────────────────────┘
```

## 一、角色与职责

### 维护者（linxin）

| 职责 | 说明 |
|---|---|
| 管理 GitHub 仓库权限 | 在 Fork 仓库添加协作者 |
| 同步官方更新 | 定期执行 upstream → main → ampere 的同步流程 |
| 审查并合并 PR | 对其他开发者的功能分支 PR 进行 Code Review |
| 维护改造日志 | 更新 `CUSTOM_CHANGES.md` |

### 开发者（所有人，包括维护者）

| 职责 | 说明 |
|---|---|
| 从 ampere 拉功能分支 | 每个功能/修复一个分支 |
| 在功能分支上开发和提交 | 不直接 push 到 ampere |
| 提交 PR 并响应审查意见 | 功能完成后提 PR，根据 Review 意见修改 |
| 保持分支最新 | 开发期间定期从 ampere 拉取最新代码 |

## 二、维护者初始设置（只做一次）

### 2.1 添加协作者

在 GitHub 上给同事开放仓库写权限，否则他们无法推送分支：

> GitHub Fork 仓库页面 → **Settings** → **Collaborators** → **Add people** → 输入同事的 GitHub 用户名 → 发送邀请

同事接受邀请后，就拥有了对 `happinessoverdue/CoPaw` 仓库的推送权限。

### 2.2 设置分支保护规则（推荐）

防止有人不小心直接 push 到 ampere（绕过 PR 流程）：

> GitHub Fork 仓库页面 → **Settings** → **Branches** → **Add branch protection rule**

| 设置项 | 建议值 | 说明 |
|---|---|---|
| Branch name pattern | `ampere` | 保护集成分支 |
| Require a pull request before merging | 勾选 | 强制通过 PR 合并 |
| Require approvals | 1 | 至少一人审批（团队小可设为 0） |
| Include administrators | 可选 | 维护者自己是否也受限 |

> **注意**：GitHub 免费版的私有仓库不支持分支保护规则，但公开仓库（Fork 出来的必然是公开的）可以用。即使不设保护规则，靠团队约定也行——约定"不直接 push 到 ampere"。

## 三、新开发者加入（每位新同事做一次）

### 3.1 克隆仓库

```bash
git clone https://github.com/happinessoverdue/CoPaw.git CoPaw-dev
cd CoPaw-dev
```

clone 会把远端所有分支的数据下载到本地，但只自动创建 `main` 分支。

### 3.2 切换到团队集成分支

```bash
git checkout ampere
```

Git 发现本地没有 `ampere`，但远端有 `origin/ampere`，会自动创建本地分支并跟踪。

### 3.3 添加官方仓库为上游（可选）

普通开发者不负责同步官方更新，这步可以不做。如果想自己也能查看官方最新代码：

```bash
git remote add upstream https://github.com/agentscope-ai/CoPaw.git
```

### 3.4 验证

```bash
git branch -a          # 查看所有分支（本地 + 远端）
git remote -v          # 查看远程仓库配置
git log --oneline -5   # 确认能看到 ampere 的提交历史
```

### 3.5 安装开发环境

```bash
conda create -n copaw-env python=3.12 -y
conda activate copaw-env
pip install -e ".[dev]"

# 构建前端（如需要）
cd console && npm ci && npm run build && cd ..
```

## 四、日常开发流程（所有开发者）

### 4.1 开始新任务：创建功能分支

```bash
# 确保在 ampere 上，并拉取最新代码
git checkout ampere
git pull origin ampere

# 从 ampere 创建功能分支
git checkout -b feat/scada-tool
```

#### 功能分支命名规范

| 前缀 | 用途 | 示例 |
|---|---|---|
| `feat/` | 新功能 | `feat/scada-tool`、`feat/alarm-skill` |
| `fix/` | 修复 bug | `fix/download-api-404` |
| `refactor/` | 重构 | `refactor/agent-hooks` |
| `docs/` | 文档 | `docs/deployment-guide` |

按**功能**命名，不按人名。一个人可以同时有多个功能分支。

### 4.2 开发与提交

```bash
# 正常开发，写代码...

# 提交（可以多次 commit，每个逻辑单元一个 commit）
git add .
git commit -m "feat: 添加 SCADA 数据读取工具"

git add .
git commit -m "feat: 添加 SCADA 工具的前端卡片渲染"
```

#### 提交消息规范

格式：`类型: 简要描述`

| 类型 | 用途 |
|---|---|
| `feat:` | 新功能 |
| `fix:` | 修复 bug |
| `refactor:` | 重构（不改变功能） |
| `docs:` | 文档变更 |
| `style:` | 代码格式（不影响逻辑） |

### 4.3 推送功能分支到 GitHub

```bash
git push origin feat/scada-tool
```

如果远端还没有这个分支，Git 会自动创建。

### 4.4 开发过程中保持与 ampere 同步

如果功能分支开发周期较长（超过几天），建议定期拉取 ampere 的最新代码，避免最终合并时冲突太多：

```bash
# 方式一：merge（推荐，简单安全）
git checkout feat/scada-tool
git merge origin/ampere

# 方式二：rebase（历史更干净，但需要对 rebase 熟练）
git checkout feat/scada-tool
git rebase origin/ampere
```

## 五、提交 PR 与代码审查

### 5.1 创建 Pull Request

功能开发完成后，在 GitHub 上创建 PR：

1. 打开 https://github.com/happinessoverdue/CoPaw
2. GitHub 通常会自动显示提示条："feat/scada-tool had recent pushes — **Compare & pull request**"，点击即可
3. 如果没有提示条：点击 **Pull requests** → **New pull request**

| 设置项 | 值 |
|---|---|
| **base** | `ampere`（合并目标） |
| **compare** | `feat/scada-tool`（你的功能分支） |
| **Title** | 简要描述这个功能，如 "feat: 添加 SCADA 数据读取工具" |
| **Description** | 详细说明改了什么、为什么、涉及哪些文件、如何测试 |

### 5.2 代码审查（Code Review）

- 维护者（或其他指定审查者）在 PR 的 **Files changed** 页面审查代码
- 可以逐行评论、提出修改建议
- 审查结果：**Approve**（通过）、**Request changes**（需要修改）、**Comment**（仅评论）

### 5.3 根据审查意见修改

如果被要求修改，在本地功能分支上继续改，然后推送：

```bash
# 根据审查意见修改代码...
git add .
git commit -m "fix: 根据 review 意见修改参数校验"
git push origin feat/scada-tool
```

PR 会自动更新，审查者能看到新的提交。

### 5.4 合并 PR

审查通过后，在 PR 页面点击 **Merge pull request**。

GitHub 提供三种合并方式：

| 方式 | 效果 | 推荐场景 |
|---|---|---|
| **Create a merge commit** | 保留功能分支所有 commit + 生成一个合并 commit | 默认选项，一般用这个 |
| **Squash and merge** | 把功能分支的所有 commit 压缩成一个 commit 合入 | 功能分支 commit 太碎时用 |
| **Rebase and merge** | 把功能分支的 commit 逐个接到 ampere 后面 | 想保持线性历史时用 |

### 5.5 合并后清理

合并完成后，GitHub 会提示 **Delete branch**，点击删除远端功能分支。

本地也清理一下：

```bash
git checkout ampere
git pull origin ampere          # 拉取合并后的最新代码
git branch -d feat/scada-tool         # 删除本地功能分支
```

## 六、维护者：同步官方更新

这部分只由维护者（linxin）操作，其他开发者不需要关心。

### 6.1 同步流程

```bash
# 1. 拉取官方最新代码
git fetch upstream

# 2. 更新 main
git checkout main
git merge upstream/main

# 3. 将官方更新合入 ampere
git checkout ampere
```

**关键决策：第 3 步用 merge 还是 rebase？**

#### 选项 A：用 merge（多人协作推荐）

```bash
git merge main
git push origin ampere
```

优点：不改写历史，其他人直接 `git pull` 就能同步，无需特殊处理。
缺点：会产生合并 commit，历史没那么"干净"。

#### 选项 B：用 rebase（历史更干净，但需要协调）

```bash
git rebase main
git push origin ampere --force-with-lease
```

**⚠️ 重要**：rebase 改写了 ampere 的提交历史，force push 后其他人的本地 ampere 就和远端对不上了。必须通知所有人执行：

```bash
git checkout ampere
git fetch origin
git reset --hard origin/ampere
```

或者（如果本地有未推送的改动）：

```bash
git checkout ampere
git pull --rebase origin ampere
```

### 6.2 同步后通知团队

无论用 merge 还是 rebase，同步完成后都应该通知团队：

> "ampere 已同步官方最新更新，请大家拉取最新代码。"
>
> 如果用了 rebase：" ampere 做了 rebase，请执行 `git fetch origin && git reset --hard origin/ampere` 更新本地分支（确保本地改动已经推送到功能分支）。"

## 七、常见场景操作手册

### 场景 1：我正在开发，维护者说 ampere 更新了

```bash
# 如果你在功能分支上，先提交或暂存当前改动
git add . && git commit -m "wip: 暂存当前进度"

# 拉取最新的 ampere
git checkout ampere
git pull origin ampere

# 回到功能分支，合入最新代码
git checkout feat/my-feature
git merge ampere
```

### 场景 2：两个人改了同一个文件，PR 有冲突

GitHub PR 页面会提示冲突无法自动合并。

**方式一：在 GitHub 上解决**（简单冲突）

点击 PR 页面的 **Resolve conflicts** 按钮，在线编辑器中手动解决。

**方式二：在本地解决**（复杂冲突）

```bash
# 在功能分支上合入最新的 ampere
git checkout feat/my-feature
git pull origin ampere

# Git 提示冲突，手动编辑冲突文件...
git add <冲突文件>
git commit -m "fix: 解决与 ampere 的合并冲突"
git push origin feat/my-feature

# PR 自动更新，冲突消失
```

### 场景 3：我想看同事的功能分支代码

```bash
git fetch origin                              # 拉取远端最新
git checkout feat/colleague-feature           # 切换到同事的分支查看
git checkout ampere                     # 看完切回来
```

### 场景 4：维护者做了 rebase + force push

**确保你功能分支上的改动已经推送到远端**，然后：

```bash
git checkout ampere
git fetch origin
git reset --hard origin/ampere          # 用远端覆盖本地
```

`reset --hard` 会丢弃本地 ampere 上未推送的改动。所以一定要确保改动在功能分支上，而不是直接在 ampere 上。

## 十、关键原则

1. **不直接 push 到 ampere** — 所有改动通过功能分支 + PR 合入
2. **不直接 push 到 main** — main 只由维护者通过 merge upstream/main 更新
3. **功能分支按功能命名，不按人名** — `feat/scada-tool` 而非 `copaw-zhang`
4. **功能分支是临时的** — 合并后及时删除，不要长期维护
5. **开发前先拉最新** — 创建功能分支前，先 `git pull origin ampere`
6. **小步快跑** — 功能拆小，尽快提 PR 合并，减少长期分叉导致的冲突

## 附录：为什么不用其他方案？

### 方案 A：所有人直接在 ampere 上开发

| 问题 | 说明 |
|---|---|
| rebase + force push 灾难 | 维护者同步官方后 force push，其他人本地分支全乱 |
| 互相踩踏 | 没有隔离，多人同时改同一区域冲突频繁 |
| 无法做代码审查 | 直接 push 没有 PR 流程 |
| 回滚困难 | 出问题时很难干净地撤回某人的改动 |

### 方案 B：每人一个长期分支（copaw-zhang），合并到 ampere

| 问题 | 说明 |
|---|---|
| 长期分支容易偏离 | 个人分支和 ampere 差异越来越大，合并代价高 |
| 一个人同时做多个功能时不灵活 | 人名分支无法区分不同功能 |
| 每人都要管 main 同步 | 没有必要，同步只需一个人做 |

### 本方案（Feature Branch + PR）

| 优点 | 说明 |
|---|---|
| 天然隔离 | 每个功能独立开发，互不影响 |
| 代码审查 | PR 流程保证代码质量 |
| 灵活并行 | 一个人可以同时开多个功能分支 |
| 干净回滚 | 某个功能有问题，revert 对应的 PR 即可 |
| 官方同步集中管理 | 只有维护者处理，其他人无感知 |
